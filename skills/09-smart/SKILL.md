# 🧠 SMART — مدیر هوشمند اسکیل‌ها

> **قرارداد با کاربر:** تو فقط SMART را صدا می‌زنی. بقیه‌اش کار من است.

SMART یک «مدیر فنی» است: هر بار فراخوانی شود، **وضعیت را می‌سنجد → تصمیم می‌گیرد → فعال می‌کند → ثبت می‌کند → نقشه‌ی راه می‌دهد**. هیچ‌وقت از کاربر نمی‌خواهد اسکیلی را دستی فعال کند.

---

```
فراخوانی SMART
│
├── گام ۰ — بازیابی هوشیاری 🔴 همیشه اول
│   ├── .project-memory/skills-state.md هست؟
│   │     ├── بله → بخوان: چه فعال است؟ صف ناتمام دارد؟ آخرین فاز؟
│   │     │        └── صف ناتمام دارد؟ → برو گام ۳ (ادامه‌ی فعال‌سازی) ⏩
│   │     └── نه → اولین اجرا است → ادامه بده
│   └── .project-memory/state.md و lessons.md هم بخوان (اگر هستند)
│
├── گام ۱ — اسکن پروژه (شواهد، نه حدس)
│   ├── ls -la ریشه + درخت فایل‌ها (عمق ۲-۳)
│   ├── چک وجود: .project-memory/ ، PLAN.md ، ARCHITECTURE.md ،
│   │            کد منبع، tests/ ، .git ، CI config ، فایل دیپلوی
│   └── اگر PLAN.md هست → وضعیت تسک‌ها را بخوان (کجای پلنیم؟)
│
├── گام ۲ — تشخیص فاز و سطح (جدول پایین)
│
├── گام ۳ — انتخاب و فعال‌سازی اسکیل‌ها
│   ├── REGISTRY.md را باز کن (کنار همین فایل)
│   ├── «بسته‌ی هوشیاری» فعال نیست؟ → همیشه اول آن (قانون ۱)
│   ├── اسکیل‌های شرط‌خورده با فاز/سطح فعلی را انتخاب کن
│   ├── قانون ضد طمع: حداکثر ۳-۴ اسکیل جدید در هر فراخوانی
│   ├── بقیه → صف انتظار در skills-state.md
│   └── هر فعال‌سازی = فوراً در skills-state.md ثبت (مقاوم به قطعی)
│
├── گام ۴ — گزارش به کاربر (قالب پایین: درختی + جدول ساده)
│
└── گام ۵ — ثبت نهایی: skills-state.md به‌روز + یک خط در worklog.md
```

## 🧭 گام ۲ — جدول تشخیص فاز (بر اساس شواهد اسکن)

| شواهد پیدا شده | فاز تشخیصی | نام فاز |
|---|---|---|
| پوشه خالی / فقط README | **P0** | 💡 کشف ایده |
| Blueprint/ایده هست، PLAN.md نیست | **P1** | 📋 برنامه‌ریزی |
| PLAN.md هست، کد اصلی نیست | **P2** | 🏗️ راه‌اندازی (اسکلت + حافظه + معماری) |
| کد هست، تسک‌های باز در PLAN | **P3** | ⚙️ توسعه‌ی فعال |
| خطا/باگ گزارش‌شده یا تست قرمز | **P3x** | 🔥 حالت دیباگ (زیرفاز P3) |
| تسک‌های فاز تمام، تحویل نزدیک | **P4** | 🧪 تثبیت و تست |
| تست‌ها سبز، نیاز به انتشار | **P5** | 🚀 دیپلوی و انتشار |
| محصول لایو، درخواست تغییر | **P6** | 🔧 نگهداری |

### سنجش سطح پروژه (تعیین می‌کند کدام لایه‌ها اصلاً لازم‌اند)

| سطح | نشانه‌ها | لایه‌های فعال از ۱۳ لایه |
|---|---|---|
| **S** کوچک | تک‌کاربره، اسکریپت/ابزار شخصی، بدون سرور | Frontend یا CLI، شاید DB ساده |
| **M** متوسط | چندکاربره، وب‌اپ با بک‌اند و DB | + API، Auth، Hosting، CI/CD، Error Tracking |
| **L** بزرگ | کاربر زیاد، درآمدی، حساس | + Security/RLS، Rate Limit، Cache/CDN، Scaling، Recovery |

⚠️ اگر سطح مبهم بود → **یک سوال کوتاه از کاربر بپرس**، حدس نزن. سطح در skills-state.md ثبت می‌شود و دیگر پرسیده نمی‌شود.

## 🛡️ قانون ۱ — بسته‌ی هوشیاری (Awareness Pack) — همیشه روشن

قبل از هر اسکیل دیگری، در **اولین فرصت** این‌ها فعال شوند و **هرگز خاموش نشوند**:

```
🛡️ بسته‌ی هوشیاری
├── 01-project-memory ← ساخت .project-memory/ (state، worklog، lessons،
│                        decisions، checkpoints، ideas-parking)
│                        → ایجنت هرگز از صفر شروع نمی‌کند
├── skills-state.md   ← حافظه‌ی خودِ SMART (چه فعال است، صف، فاز، سطح)
└── 06-git-discipline ← هر تغییر کامیت؛ checkpoint = نقطه‌ی برگشت امن
```

**نتیجه:** اگر ایجنت وسط کار قطع شد، در فراخوانی بعدیِ SMART:
«تا استپ چند رفتیم» ← `state.md` | «چه خطاهایی داشتیم» ← `lessons.md` | «چه ناتمام ماند» ← `worklog.md` + صف skills-state | «آخرین نقطه‌ی سالم» ← `checkpoints.md` + تگ git

> اگر پروژه اسکیل‌های حافظه‌ی قوی‌تری در دسترس دارد (مثل `agentdb-memory-patterns`)، در سطح M/L به‌عنوان **تقویت** روی همین پایه فعال کن — جایگزینِ فایل‌های حافظه نمی‌شود، مکملش است.

## 🎯 گام ۳ — ماتریس فعال‌سازی (خلاصه؛ جزئیات کامل در REGISTRY.md)

```
P0 کشف ایده      → 🛡️ هوشیاری + 08-project-idea-architect
P1 برنامه‌ریزی    → 🛡️ + 02-task-planner (+ sparc-methodology اگر M/L)
P2 راه‌اندازی     → 🛡️ + 03-architecture-mapper + 04-clean-code
P3 توسعه         → 🛡️ + 04-clean-code + 02 (پیگیری پلن)
                    ├─ سطح M/L: + pair-programming یا verification-quality
                    └─ کار UI دارد: + browser
P3x دیباگ        → 🛡️ + 05-debug-protocol (همه‌ی فعال‌سازی جدید متوقف!)
P4 تثبیت         → 🛡️ + 07-qa-testing (+ performance-analysis اگر کندی هست)
P5 دیپلوی        → 🛡️ + 07 + github-workflow-automation
                    (+ github-release-management اگر M/L)
P6 نگهداری       → 🛡️ + 05 + github-project-management (+ code-review اگر PR-محور)
```

### ⛔ قوانین انتخاب

1. **حداقل‌گرایی:** فقط اسکیلی که *همین فاز* لازم دارد. «شاید بعداً» → صف انتظار با یادداشت «بعد از فاز X».
2. **سقف فعال‌سازی:** حداکثر ۳-۴ اسکیل جدید در هر فراخوانی. بیشتر لازم بود → صف؛ کاربر دوباره SMART را صدا می‌زند و ادامه می‌دهی.
3. **سازگاری با سطح:** اسکیل 🔴 (سنگین) فقط برای سطح L. اسکیل ⚫ (داخلی ruflo مثل v3-*, flow-nexus-*) **هرگز**.
4. **تشخیص فریب اسم:** `agentdb` = حافظه‌ی *ایجنت* است نه دیتابیس *پروژه*! برای DB پروژه اسکیل DB فعال نکن مگر واقعاً موجود و مرتبط باشد.
5. **حالت دیباگ = قفل:** در P3x هیچ اسکیل جدیدی فعال نمی‌شود تا مشکل حل شود (جلوی فرار به جلو).
6. **غیرفعال‌سازی هم کار توست:** اسکیلی که فازش گذشته (مثلاً 08 بعد از P0) را در skills-state به حالت `standby` ببر تا شلوغی ذهنی ایجاد نکند.

## 🔌 پروتکل قطعی و ادامه (Resume)

```
قطع شد؟ کاربر فقط دوباره SMART را صدا می‌زند:
│
├── skills-state.md → صف ناتمام دارد؟ ادامه‌ی فعال‌سازی از همان‌جا
├── state.md → «آخرین بار اینجا بودیم، کار بعدی این بود»
├── worklog.md → آخرین ردیف = آخرین کار (تمام یا ناتمام؟)
├── lessons.md → خطاهای باز (⚠️ حل‌نشده‌ها اول)
└── git log/tag → آخرین نقطه‌ی سالم برای برگشت
     └── گزارش: «قطع شدیم در X؛ ادامه از Y» — نه شروع از صفر!
```

## ⛔ ممنوعیات SMART

- فعال‌سازی اسکیل بدون ثبت در skills-state.md
- فعال‌سازی ⚫ (داخلی ruflo: v3-*, flow-nexus-*, worker-benchmarks)
- بیش از ۴ اسکیل جدید در یک اجرا
- حدس‌زدن فاز بدون اسکن واقعی فایل‌ها
- خاموش‌کردن بسته‌ی هوشیاری — تحت هیچ شرایطی
- اسکیل جدید وسط حالت دیباگ (P3x)
