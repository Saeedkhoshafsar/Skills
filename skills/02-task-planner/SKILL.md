# Task Planner — پلن و ترتیب کارها

ذهنیت: **مدیر پروژه‌ی سخت‌گیر و واقع‌بین.** هدف: کاربر هیچ‌وقت وسط آشفتگی رها نشود،
چون هر فاز به‌تنهایی یک خروجی سالم و قابل‌استفاده است.

## 🧱 اصول فازبندی

1. **قانون MVP سخت‌گیرانه:** فاز ۱ = کوچک‌ترین نسخه‌ای که *کار می‌کند* و *قابل نمایش* است.
2. **هر فاز ≤ یک جلسه‌ی کاری.** اگر بزرگ‌تر است، بشکن.
3. **هر فاز با فاز قبلی جمع می‌شود، نه بازنویسی.** (incremental, not rewrite)
4. **هر تسک اتمیک است:** یک کار، یک نتیجه‌ی قابل‌تست، یک کامیت.
5. **وابستگی‌ها صریح نوشته می‌شوند.** هیچ تسکی قبل از پیش‌نیازش شروع نمی‌شود.
6. **قابلیت‌های «شاید بعداً» → `ideas-parking.md`.** وارد پلن نمی‌شوند.

## 📄 خروجی الزامی: فایل `PLAN.md` در ریشه‌ی پروژه

```markdown
# پلن پروژه: [نام]

## هدف نهایی (یک جمله)
[چه مشکلی برای چه کسی حل می‌شود]

## فازها

### Phase 1 — MVP: [عنوان]  🎯 خروجی: [چیز قابل نمایش]
| # | تسک | پیش‌نیاز | معیار انجام‌شدن (DoD) | وضعیت |
|---|---|---|---|---|
| 1.1 | راه‌اندازی اسکلت پروژه | — | `npm run dev` بالا می‌آید | ⬜ |
| 1.2 | مدل داده + مایگریشن | 1.1 | جدول‌ها ساخته می‌شوند | ⬜ |
| 1.3 | CRUD پایه | 1.2 | تست دستی ۴ عملیات OK | ⬜ |

### Phase 2 — [عنوان]  🎯 خروجی: [...]
...

## خارج از محدوده (فعلاً نه!)
- [قابلیت X] → ideas-parking
```

## ✅ معیار انجام‌شدن (Definition of Done) — برای هر تسک

- [ ] کد نوشته و مطابق `04-clean-code` است
- [ ] به‌صورت واقعی اجرا/تست شده (نه فقط «باید کار کنه»)
- [ ] کامیت شده (`06-git-discipline`)
- [ ] در `worklog.md` ثبت شده

## 🚦 ترتیب اجرا در هر جلسه

1. `state.md` را بخوان → فاز و تسک فعلی را پیدا کن
2. فقط **یک تسک** را در لحظه انجام بده (تک‌تمرکزی)
3. تسک تمام شد → وضعیت در `PLAN.md` → ✅، ثبت در worklog، کامیت
4. فاز تمام شد → `07-qa-testing` → سبز شد → checkpoint بزن

## ⛔ علائم هشدار (فوراً متوقف شو)

- تسکی که ۲ برابرِ تخمین طول کشیده → بشکنش یا به `05-debug-protocol` برو
- وسوسه‌ی «حالا که اینجاییم، فلان چیز رو هم اضافه کنیم» → پارکینگ ایده‌ها
- فازی که خروجی قابل نمایش ندارد → فازبندی غلط است، بازطراحی کن
