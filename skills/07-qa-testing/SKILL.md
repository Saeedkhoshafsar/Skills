# QA & Testing — کنترل کیفیت

ذهنیت: **QA بدبین.** فرض پیش‌فرض: کد خراب است، تا وقتی با اجرای واقعی خلافش ثابت شود.

## 🥇 قانون طلایی

> **«تمام شد» فقط یعنی: اجرا کردم، با چشم خودم دیدم، نتیجه را ثبت کردم.**

«کد نوشتم و منطقاً باید کار کند» = تمام نشده.

## 🧪 سطوح تست (به تناسب پروژه)

### سطح ۱ — تست دودی (Smoke) — همیشه، بعد از هر تسک
- سرویس/برنامه بالا می‌آید؟ بدون خطا در لاگ استارت؟
- مسیر اصلی همان تسک را **واقعاً اجرا کن** (curl / مرورگر / اجرای اسکریپت) و خروجی واقعی را ببین

### سطح ۲ — تست‌های خودکار — برای منطق مهم
- منطق کسب‌وکار (services) و توابع خالص utils تست واحد داشته باشند
- هر باگِ حل‌شده → یک تست که دیگر برنگردد (regression test)
- تست‌ها در پوشه‌ی `tests/` آینه‌ی ساختار `src/`

### سطح ۳ — تست مسیرهای بد — قبل از بستن هر فاز
- ورودی خالی / null / نوع اشتباه
- ورودی خیلی بزرگ / کاراکترهای خاص / یونیکد (متن فارسی!)
- سرویس وابسته در دسترس نیست (DB قطع، API خارجی timeout)
- دو درخواست هم‌زمان روی یک منبع

## 📋 چک‌لیست تحویل فاز (قبل از checkpoint)

- [ ] همه‌ی تسک‌های فاز در `PLAN.md` تیک خورده‌اند
- [ ] تست دودی کل مسیرهای فاز — با خروجی واقعیِ دیده‌شده
- [ ] تست‌های خودکار (اگر هستند) همه سبز — خروجی اجرا در worklog ثبت شود
- [ ] حداقل ۳ مسیر بد امتحان شد و رفتار معقول بود (خطای مفهوم، نه crash)
- [ ] هیچ خطا/warning جدیدی در کنسول و لاگ نیست
- [ ] کد دیباگ موقت (print/console.log) پاک شده
- [ ] `ARCHITECTURE.md` و `PLAN.md` با واقعیت هم‌خوانند
- [ ] کامیت + تگ checkpoint (`06-git-discipline`)

## 🧾 گزارش تست (در worklog)

```markdown
### تست فاز 1 — 2026-07-06
- ✅ استارت سرور: بدون خطا (پورت 3000)
- ✅ POST /api/login با کاربر معتبر → 200 + توکن
- ✅ POST /api/login با پسورد غلط → 401 با پیام مفهوم
- ✅ ورودی خالی → 400 (نه crash)
- ⚠️ متن فارسی در فیلد نام: ذخیره OK، ولی نمایش ؟؟؟ → ثبت شد به‌عنوان تسک 2.4
```

## ⛔ ممنوعیات

- اعلام موفقیت بدون اجرای واقعی
- پاک‌کردن یا کامنت‌کردن تستِ قرمز برای «سبز شدن»
- تست فقط happy path
- پنهان‌کردن warning ها («فعلاً کار می‌کنه») — یا حل کن یا آگاهانه در lessons ثبت کن
